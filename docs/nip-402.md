NIP-402
======

Paid HTTP Service Announcements
-------------------------------

`draft` `optional`

This NIP defines kind `31402` (a parameterized replaceable event) for advertising paid HTTP services on Nostr. Services publish what they offer, what it costs, and how they accept payment. Clients discover services by topic, capability, or payment method via relay filters.

This NIP covers discovery and capability advertising only. Payment flow is delegated to the underlying payment protocol (L402, x402, NWC, etc.) and is explicitly out of scope.

## Motivation

Paid HTTP APIs are everywhere, but there is no decentralized way to discover them. Developers publish docs on websites, list on marketplaces, or rely on word of mouth. AI agents have no machine-readable catalogue of services they can pay for and consume autonomously.

[NIP-90](90.md) Data Vending Machines solve a related but different problem: Nostr-native compute jobs where the request and result flow through relays. Many useful services are conventional HTTP APIs that accept a request and return a response. These services need discovery but not a new execution model.

Kind `31402` gives any paid HTTP endpoint a Nostr-native presence - discoverable through relay filters, verifiable through [NIP-05](05.md), and machine-readable enough for both human clients and AI agents to find, evaluate, and consume services without prior knowledge of them.

## Event Kind

Kind `31402` - parameterized replaceable event ([NIP-33](33.md)).

`30000 + 1402` references HTTP `402 Payment Required`.

Relays store exactly one event per `pubkey` + `d` tag combination. Re-publishing with the same `d` tag replaces the previous listing.

## Tags

| Tag | Format | Required | Purpose |
|-----|--------|----------|---------|
| `d` | `["d", "<identifier>"]` | Yes | Unique listing ID per pubkey ([NIP-33](33.md)). Free-form string. |
| `name` | `["name", "<display-name>"]` | Yes | Human-readable service name |
| `url` | `["url", "<endpoint>"]` | Yes (1+) | Service endpoint URL. Repeatable for multi-transport. |
| `about` | `["about", "<description>"]` | Yes | Short description of what the service does |
| `pmi` | `["pmi", "<method>"]` | Yes (1+) | Payment method identifier. Repeatable. |
| `price` | `["price", "<capability>", "<amount>", "<currency>"]` | No | Per-capability pricing. Repeatable. |
| `t` | `["t", "<topic>"]` | No | Topic tags for relay-side filtering. Repeatable. |
| `picture` | `["picture", "<url>"]` | No | Service icon or logo URL |

### Tag Details

**`url` tag:** The base URL of the service. Clients append capability-specific paths as needed. For single-endpoint services, this is the full endpoint URL. The `url` tag is repeatable; services MAY publish multiple `url` tags to advertise the same service over different transports (see [Multi-Transport Endpoints](#multi-transport-endpoints)).

**`price` tag:** Amounts are string integers in the smallest currency unit. Capability names MUST match a corresponding `capabilities[].name` in the content JSON when both are present.

**`pmi` tag:** Payment method identifiers are extensible strings. Services MUST include at least one. Recommended values:

- `bitcoin-lightning-bolt11` - Lightning BOLT-11 invoices
- `bitcoin-cashu` - Cashu ecash tokens
- `x402-usdc-base` - x402 USDC on Base
- `x402-usdc-ethereum` - x402 USDC on Ethereum

Any string is valid. Clients filter by payment methods their wallet supports.

### Currency Identifiers

Recommended currency strings for the `price` tag:

| Identifier | Unit | Description |
|------------|------|-------------|
| `sats` | satoshi | Bitcoin (Lightning, on-chain) |
| `msats` | millisatoshi | Bitcoin (sub-satoshi precision) |
| `usd` | cent | US Dollar |
| `eur` | cent | Euro |

Implementations SHOULD use these identifiers for interoperability. Custom currency strings are permitted.

## Content

The event content is a JSON string with optional extended metadata:

```json
{
  "capabilities": [
    {
      "name": "standard-joke",
      "description": "Solid jokes across 6 topics",
      "endpoint": "/api/joke?tier=standard",
      "schema": {},
      "outputSchema": {}
    }
  ],
  "version": "1.0.0"
}
```

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `capabilities` | array | No | List of capabilities with optional JSON schemas |
| `capabilities[].name` | string | Yes | Capability identifier (matches `price` tag capability) |
| `capabilities[].description` | string | No | Human-readable description |
| `capabilities[].endpoint` | string | No | Endpoint path or full URL for this capability |
| `capabilities[].schema` | object | No | JSON Schema for request body |
| `capabilities[].outputSchema` | object | No | JSON Schema for response body |
| `version` | string | No | Service version |

All essential discovery data lives in tags so relays can filter on it. Content is supplementary. If no extended metadata is needed, content SHOULD be `{}` (empty JSON object).

Capability names in the content JSON MUST match the corresponding `price` tag capability names when both are present. Capabilities MAY appear in content without a `price` tag (free capabilities) or in `price` tags without content detail (priced but undescribed).

**`endpoint` field:** When present, `endpoint` specifies where to reach this capability. It MUST be either a relative path (starting with `/`) or a full URL. Relative paths are resolved against the `url` tag (when multiple `url` tags are present, the relative path applies to each). When absent, clients should use the `url` tag directly. This field enables services with multiple API endpoints to advertise each one precisely, avoiding the need for clients to guess or probe for paths.

JSON schemas are optional but recommended. When present, clients can construct valid requests from the announcement without probing the endpoint.

## Multi-Transport Endpoints

A service MAY publish multiple `url` tags to advertise the same service over different network transports. This enables censorship resistance: if one transport is seized, blocked, or revoked (DNS seizure, registrar suspension, CA revocation), clients fall back to an alternative without losing access to the service.

### Transport Types

| Transport | Example | Notes |
|-----------|---------|-------|
| HTTPS | `https://api.example.com` | Standard clearnet; recommended as baseline |
| Tor | `http://exampleonion...onion` | Onion v3 hidden service; requires SOCKS proxy |
| Handshake | `https://api.example` | HNS top-level domain; resolved via DNS-over-HTTPS gateway |
| HTTP | `http://api.example.com` | Plain HTTP; use only where TLS is unavailable (e.g. onion) |

Clients classify URLs by hostname and scheme:
- Hostname ending in `.onion` - **onion**
- Hostname with a TLD not in the ICANN root zone - **hns** (heuristic)
- `https://` with a standard TLD - **https**
- `http://` with a standard TLD - **http**

### Example Event (Multi-Transport)

```json
{
  "tags": [
    ["d", "my-service"],
    ["name", "My Service"],
    ["url", "https://api.example.com"],
    ["url", "http://abc123...onion"],
    ["url", "https://api.exampleservice"],
    ["about", "Same service, three transports"],
    ["pmi", "bitcoin-lightning-bolt11"]
  ]
}
```

All `url` tags point to the same logical service. Clients pick whichever transport they can reach.

### Client Behaviour

Clients SHOULD:

1. **Filter** by capability - remove transports the client cannot use (e.g. `.onion` without a Tor proxy).
2. **Sort** by preference - privacy-maximising clients prefer `onion > hns > https > http`; latency-sensitive clients may prefer `https > hns > onion`.
3. **Fallback** on transport failure - connection-level errors (DNS failure, ECONNREFUSED, timeout, TLS handshake failure) trigger fallback to the next URL. HTTP-level responses (including 402, 500) do NOT trigger fallback.

### Credential Keying

When a service is identified by its Nostr pubkey (from the kind `31402` event), clients SHOULD key cached credentials (macaroons, tokens, session state) by **pubkey**, not by URL origin. This allows a credential obtained via one transport to be reused over another without re-authentication.

### Multiple URLs vs Multiple Events

Multiple `url` tags in a **single event** represent the same service on different transports. Separate kind `31402` events (different `d` tags) represent genuinely different services from the same operator.

## Discovery

### Relay-Side Filtering

Clients SHOULD use relay filter syntax to reduce bandwidth:

```json
{"kinds": [31402], "#t": ["jokes"]}
{"kinds": [31402], "#pmi": ["bitcoin-lightning-bolt11"]}
{"kinds": [31402], "#pmi": ["bitcoin-cashu", "bitcoin-lightning-bolt11"]}
```

### Client-Side Filtering

After relay filtering, clients MAY further narrow results by:
- Text search across `name`, `about`, capability names and descriptions
- Payment method matching against wallet capabilities
- [NIP-05](05.md) trust verification (see below)
- `created_at` age as a freshness heuristic

Services SHOULD re-publish periodically (e.g. every 24 hours) to signal liveness. Clients MAY discard listings with a `created_at` older than a reasonable threshold.

## Trust and Security

### Domain Verification

Implementations SHOULD verify that the pubkey's [NIP-05](05.md) identifier domain matches at least one `url` tag domain. This provides an anti-spoofing signal: a service claiming to be at `https://example.com` should have a NIP-05 on `example.com`.

For multi-transport services, NIP-05 verification applies to the clearnet (`https://`) URL. Onion and HNS endpoints are trusted transitively - they share the same Nostr pubkey as the verified clearnet endpoint.

Clients MAY maintain whitelists or reputation scores for trusted service providers.

### Threat Considerations

- **Spoofing:** An attacker could publish a kind `31402` event pointing to a URL they do not control, hoping to intercept payments. NIP-05 domain verification mitigates this. Clients SHOULD warn users when domain verification fails.
- **Stale listings:** A service may go offline without deleting its announcement. Clients SHOULD treat old `created_at` timestamps as a signal of potential staleness and MAY probe the `url` before presenting it to users.
- **Price manipulation:** Announcements are self-reported. Clients SHOULD treat `price` tags as indicative and confirm actual pricing with the service at request time.
- **Relay spam:** Relays MAY rate-limit kind `31402` events per pubkey to prevent catalogue flooding.

## Relationship to Other NIPs

- **[NIP-90](90.md) (DVMs):** Complementary. DVMs handle Nostr-native compute jobs (intent-based, relay-mediated). This NIP handles HTTP API discovery (offer-based, out-of-band execution). A service MAY publish both a DVM kind and a kind `31402` event if it supports both interaction models.
- **[NIP-89](89.md) (App Handlers):** Orthogonal. NIP-89 routes unknown event kinds to handler apps. This NIP advertises paid HTTP services.
- **[NIP-33](33.md):** This NIP uses parameterized replaceable events.
- **[NIP-05](05.md):** Recommended for trust verification (domain matching).
- **[NIP-98](98.md) (HTTP Auth):** Complementary. NIP-98 authenticates Nostr users to HTTP servers. This NIP advertises HTTP services for discovery. A paid service could use NIP-98 for identity and kind `31402` for discoverability.
